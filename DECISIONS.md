# DECISIONS.md

## 1. 상태 구조

- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
  - 서버에서 가져온 태스크 목록과 UI 상태를 React로 관리함(useState, useRef, useMemo, useCallback)
  - `useTaskFetch`에서 서버 데이터 조회 및 관리
  - `useTaskMutation`에서 생성/수정/삭제/이동 등 상태 변경 로직 관리
  - 컴포넌트 간 상태 공유는 props 전달 방식으로 처리함
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?
  - 전역 상태 관리 라이브러리 없이도 상태 흐름을 명확하게 관리 가능하다고 판단함
  - 서버 상태와 UI 상태 변경 로직을 hook 단위로 분리하여 유지보수성 확보
  - Context API: props 전달 범위가 깊지 않아 Context 도입의 필요성이 낮았음
  - React Query: 서버 상태 관리에는 적합하지만, Optimistic Update와 rollback, conflict 처리 로직을 직접 제어하기 위해 사용하지 않음

<br/>

## 2. 낙관적 업데이트 & 롤백

- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
  - 요청 실패 시, 이전 상태로 rollback 처리함
  - 태스크 이동/수정/삭제 시 optimistic update 적용
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?
  - 이전 스냅샷 복원 방식 사용
  - 실패 시, 서버 재조회 방식은 추가 네트워크 요청이 필요하고, 실패 직전 사용자의 변경 상태를 정확히 복원하기 어려움

<br/>

## 3. 경쟁 상태

- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  - 요청 순번 비교 방식을 사용함
  - 같은 태스크에 대해 빠른 연속 수정이 발생할 경우 이전 요청의 늦은 응답이 최신 상태를 덮어쓰는 문제 방지 필요
    - 요청마다 증가하는 `requestSeq` 생성
    - 태스크별 최신 요청 ID를 `latestRequest` Map으로 관리
    - 응답 도착 시, 현재 최신 요청인지 확인 후 상태 반영
    - 오래된 요청 응답은 무시하고, 가장 마지막 사용자 요청 상태만 반영함

<br/>

## 4. 대량 데이터 성능

- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
  - 현재 구조에서는 컴포넌트 렌더링 최적화를 우선 적용함
  - `Column`, `Card` 컴포넌트에 React.memo 적용
  - useMemo를 사용하여 상태별 태스크 목록을 메모이제이션함
  - 가상 스크롤 미적용
    - 5,000개 데이터 렌더링 개선 효과는 있지만, Drag & Drop과 optimistic update 상태 관리 구조를 함께 고려해야 함
    - 현재 요구사항의 핵심인 데이터 정합성과 상태 관리 안정성을 우선 구현함

   <br/>

## 5. 정답이 없던 결정들

- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
  - 요청 실패 시, Optimistic Update 상태를 rollback 처리하고 사용자에게 실패 메시지를 안내함
  - 서버 반영 성공 여부를 기준으로 상태를 관리하는 것이 안전하다고 판단함
- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
  - 서버의 최신 데이터를 우선 반영하고, 사용자에게 충돌 상황을 안내함
  - version 기반 충돌 감지를 통해 최신 서버 상태 확인
  - 충돌 발생 시, 클라이언트 변경사항보다 서버의 최신 데이터가 신뢰할 수 있는 상태라고 판단함
- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?
  - 자동 재시도 대신 사용자가 직접 다시 시도하는 방식을 선택함
  - 사용자가 실패 상황을 확인 후, 재요청하는 방식이 안전하다고 판단함

<br/>

## 6. 그 외 트레이드오프 / 남은 한계

- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.
  - 검색 디바운싱, 우선순위/상태/태그 다중 필터
    - 태스크 상태 변경, 서버 동기화, 실패 처리 등 핵심 요구사항 구현을 우선함
  - 가상 스크롤 적용
    - 대량 데이터 렌더링 성능 개선을 위해 react-window 등 가상화 라이브러리 도입 고려

<br/>

  - 과제 범위 내에서 복잡한 UI 컴포넌트가 필요하지 않았음
